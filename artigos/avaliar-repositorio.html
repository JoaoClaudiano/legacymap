Workshop Prático de 60 Minutos

Siga este workshop passo a passo para, em apenas 60 minutos, gerar um relatório completo sobre a saúde arquitetural de qualquer projeto open source no GitHub. Perfeito para avaliar bibliotecas antes de adotá-las ou para auditorias rápidas em projetos internos.

TEMPO TOTAL DO WORKSHOP

60 MINUTOS

Dividido em 4 fases de 15 minutos cada

Minuto 0-15: Configuração e Primeira Análise

1
Escolha um repositório alvo

Selecione um repositório público no GitHub para análise. Sugestões:

  • Um framework JavaScript que você utiliza (React, Vue, Angular)
  • Uma biblioteca que você está considerando adotar
  • Um repositório interno da sua organização (se disponível publicamente)
2
Cole a URL no CodeCartographer

Abra o CodeCartographer em seu navegador e cole a URL completa do repositório GitHub no campo de análise.

Dica profissional:

Para uma análise mais rápida, prefira repositórios com menos de 10.000 arquivos. Projetos muito grandes podem exigir mais tempo de processamento.

3
Gere o mapa arquitetural completo

Clique em "Analisar" e aguarde alguns segundos enquanto o CodeCartographer processa todas as dependências do projeto e gera o mapa visual interativo.

4
Salve a imagem inicial como referência

Use a funcionalidade de exportação para salvar uma imagem do mapa arquitetural inicial. Isso será útil para comparação após possíveis refatorações.

Minuto 15-30: Análise dos Padrões Arquiteturais

Nesta fase, vamos investigar os padrões arquiteturais presentes no projeto. Responda às seguintes perguntas:

Existem hubs óbvios? (módulos com +20 conexões)

Identifique módulos com um número anormalmente alto de dependências. No CodeCartographer, esses aparecem como nós maiores ou com mais conexões.

Há ciclos de dependência?

Use o detector automático de ciclos do CodeCartographer para identificar dependências circulares entre módulos. Anote quaisquer ciclos encontrados.

Qual a distribuição de acoplamento?

Analise se a maioria dos módulos tem 1-2 conexões (saudável) ou se há muitos módulos altamente acoplados (potencial problema).

Tarefa prática

Crie uma tabela com:

  • Número total de módulos identificados
  • Quantidade de hubs (+20 conexões)
  • Ciclos detectados (sim/não e quantos)
  • Distribuição de acoplamento (baixo, médio, alto)

Minuto 30-45: Investigação Profunda e Contexto

Agora vamos mergulhar mais fundo nos pontos críticos identificados:

1
Selecione o módulo mais conectado

Clique no módulo com maior número de conexões para ver suas dependências detalhadas.

2
Analise sua função

Determine se é um "coordinator" legítimo (como um roteador ou gerenciador de estado) ou um "god object" (classe/módulo que faz demasiado).

Como diferenciar:

Um coordinator legítimo geralmente delega trabalho para outros módulos. Um god object contém lógica que deveria estar distribuída.

3
Verifique sua evolução histórica

No GitHub, examine o histórico de commits desse módulo. Ele cresceu organicamente ao longo do tempo ou foi planejado assim desde o início?

4
Repita para outros pontos de interesse

Se houver tempo, analise também o segundo e terceiro módulos mais conectados para identificar padrões.

Minuto 45-60: Relatório Final e Insights Acionáveis

Com base na sua análise, crie um relatório conciso com as seguintes seções:

1. Pontos fortes da arquitetura

Liste 2-3 aspectos positivos identificados. Exemplos:

  • Separação clara de responsabilidades entre camadas
  • Baixo acoplamento entre módulos de negócio
  • Ausência de ciclos de dependência críticos
2. 3 principais oportunidades de melhoria

Identifique problemas arquiteturais com maior impacto. Priorize com base em:

  • Risco de manutenção
  • Impacto no desempenho
  • Dificuldade para novos desenvolvedores
3. 1 recomendação prioritária

Escolha a ação mais importante que os mantenedores deveriam tomar. Seja específico:

Exemplo de recomendação:

"Refatorar o módulo UserService para extrair a lógica de validação em um Validator dedicado, reduzindo suas dependências de 28 para aproximadamente 12."
